Research-with-context vs. vanilla agents

Does grounding an agent in these lecture notes beat an off-the-shelf frontier model at doing real research work?

A benchmark design - not yet run. Contributions welcome.

The question

Frontier assistants from Anthropic, OpenAI and others already do “research” out of the box: they read papers, summarise, and propose ideas. The claim worth testing is narrower and more useful: if you first feed an agent a compact, well-structured methodology - the CS197 notes in this reader - does it do measurably better research work than the same model with no such grounding?

The notes encode process, not facts: how to read wide then deep, how to build a problem–solution chain, how to keep a “to understand” list, how to structure a paper. That's exactly the kind of tacit skill a base model is uneven at. So this is a test of whether cheap context can substitute for it.

Hypothesis

An agent given the CS197 methodology as context produces literature reviews and research plans that are more structured, better sourced, and more honest about gaps than the same model run vanilla - at a small, fixed token cost - and the gap widens on unfamiliar topics.

Conditions

Hold the model fixed and vary only the context. Run each condition on the same model (e.g. Claude and a GPT model), so differences are attributable to context, not model choice.

  • A - Vanilla: the model with a plain instruction (“research topic X and write a literature review / research proposal”). No extra context. This is the “straight from the AI company” baseline.
  • B - Notes in context: the same prompt, preceded by the relevant lecture Markdown from /md (e.g. Lecture 3 for a literature-review task).
  • C - Notes + tools: condition B, plus web/search and PDF-reading tools, so the methodology can actually drive real retrieval.
  • D - Ablation: notes with the process stripped out (only the topic facts left), to check that any gain comes from methodology rather than from extra tokens.

Tasks

Use topics the model is unlikely to have deep priors on, so grounding can show. For each topic, three deliverables:

  1. Literature map - identify the key methods, datasets, benchmarks and one survey; summarise each in a few bullets with links (the Lecture 3 “reading wide” output).
  2. Deep read - pick one paper, extract its problem–solution chain, and produce a “to understand” list.
  3. Research proposal - a one-page, testable idea with a method sketch and an evaluation plan.

What to measure

  • Citation validity - fraction of cited papers that exist and actually support the claim (catches hallucinated references).
  • Coverage - recall of the genuinely important works for the topic, scored against an expert-built key set.
  • Structure - does the output contain the methodology's artifacts (problem–solution chain, to-understand list, wide-then-deep flow)?
  • Calibration / honesty - does it flag what it doesn't know, instead of confabulating?
  • Cost - tokens and wall-clock, so any quality gain is judged against its price.

Scoring

Blind the grader to the condition. Use a rubric (1–5 per metric) applied by a human expert and, separately, by an LLM judge given the rubric; report both and their agreement. Randomise task order; run ≥3 seeds per cell and report variance, not just means.

How to run it with this reader

The Markdown files are the whole point: each lecture's canonical source is a plain .md under /md, so an agent can fetch it directly.

# condition B input = notes + task
cat md/lecture-03.md > context.md
# then prompt your model:
#   [context.md]
#   Task: for the topic "<X>", produce a literature map, a deep read,
#   and a one-page research proposal, following the methodology above.

Point the agent at http://localhost:8000/md/lecture-03.md (while the reader's local server is running) or hand it the file. Keep the vanilla condition's prompt byte-identical minus the context block.

Threats to validity

  • Contamination: the CS197 notes and the example papers are public and may be in pretraining. Prefer recent topics and hold-out papers; the ablation (D) helps separate methodology from memorised content.
  • Token confound: condition B adds tokens. D controls for “more context helps regardless of what it is.”
  • Judge bias: an LLM judge may prefer verbose, structured text. Anchor with a human expert on a subsample and report agreement.
  • Topic luck: few topics = high variance. Use enough topics to separate signal from noise.

Status

This is a design, not a result. It is the reason the reader is Markdown-first: the content doubles as an agent's context pack. Lecture 3 is ready to use as the grounding for a literature-review run today; the remaining lectures are being transcribed.
